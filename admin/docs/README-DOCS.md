# The portfolio docs hub — how this works

A read-only, browsable knowledge base for **all the projects in the portfolio**,
rendered by the `/admin/` docs board. This file is the map.

## What this is (and isn't)

- **Is:** PRDs, roadmaps, decisions, and notes for each project (Aleph, Spar,
  Fantastic Leagues, …), organized so future-me (and the board) can find things.
- **Isn't:** documentation of jameschang.co's own code — that lives in the existing
  `/docs/` knowledge base (`docs/solutions/`, `docs/guides/`). The hub *links* to
  those; it doesn't duplicate them.

## Public-repo reality (important)

This repo is public and `/admin/` is a curtain, not a lock. **Everything written here
is effectively public.** So the same rule as the portfolio board applies: hub content
is **public-safe only** — professional-candid PRDs and roadmaps, no secrets, no
credentials, no competitive material you wouldn't want a peer to read. Anything truly
private stays out of this repo entirely.

## Layout

```
admin/docs/
  README-DOCS.md        ← this file
  INBOX.md              ← open comments (regenerated by bin/sync-inbox.py — Step 5)
  _comments.json        ← local comment data for the inbox loop
  _templates/           ← prd / adr / doc starting points (Step 8)
  foundations/          ← cross-portfolio: glossary, feature-intake rules, conventions
  projects/
    <slug>/             ← one folder per project (matches the /admin board + config)
      prds/  roadmap.md  adrs/  notes.md
```

## Frontmatter convention (Step 2)

Every **authored** doc opens with this YAML block. It is the single source of truth
the board reads to index, tag, filter ("show me all Aleph PRDs that are still draft"),
and cross-link. No frontmatter → the board can't see the doc.

```yaml
---
id: PRD-001          # stable ID, never reused. PRD-### | ADR-### | DOC-### |
                     #   RISK-### | EXP-###  (numbers run per type)
type: prd            # see the type list below
project: aleph       # slug from bin/projects-config.json (aleph, spar, …),
                     #   OR "portfolio" for cross-project docs,
                     #   OR "jameschang-co" for this site itself
status: draft        # draft | active | locked | done | deprecated
owner: james         # who owns this doc
tags: []             # ONLY from the controlled vocab below — never freeform
links: []            # IDs of related docs (e.g. [ADR-002, PRD-004]) — traceability
updated: 2026-07-23  # YYYY-MM-DD, bumped on every edit
phase: null          # optional: which build phase, or null
---
```

**`type`** (what kind of doc it is — drives which board section it lands in):
`prd` · `roadmap` · `todos` · `adr` · `tech-spec` · `api-docs` · `decision-log` ·
`testing` · `component-lib` · `changelog` · `risk` · `experiment` · `privacy` ·
`runbook` · `glossary` · `intake-rules` · `launch-spec` · `stats` · `costs` ·
`status` · `note`

**`status`** — `draft` (being written) · `active` (current, in use) · `locked`
(frozen, changes need the intake process) · `done` (shipped/closed — surfaced as a
filter, the doc is NOT moved to another file) · `deprecated` (kept for history).

### Controlled tag vocabulary

Tags capture **cross-cutting themes** that span projects — the slices you'd want to
compare across the whole portfolio (project is already its own field, so tags are NOT
project names). **The list is closed:** if a doc needs a tag that isn't here, add it to
this list deliberately — never invent a one-off, or search rots into noise.

| tag | use it for |
|---|---|
| `ai` | AI / LLM-powered features (Aleph checks, Spar counterpart, FL insights) |
| `voice` | voice / audio experiences (Spar, KTV Singer) |
| `payments` | Stripe, billing, monetization |
| `auth` | login, accounts, permissions |
| `onboarding` | first-run, activation, setup flows |
| `data-model` | schema, entities, migrations |
| `integration` | third-party APIs and data feeds |
| `ui-ux` | interface, design system, interaction |
| `mobile` | iOS / Android / React Native apps |
| `infra` | hosting, deploy, ops, cron |
| `growth` | acquisition, retention, marketing |
| `compliance` | regulatory / safety (Aleph FSVP·CPSIA·Prop65, Vouch minor-safety) |

## Other conventions (later steps — placeholders for now)

- **IDs** — stable per type: `PRD-###`, `ADR-###`, `DOC-###`, `RISK-###`, `EXP-###`.
  Assigned once, never reused, even if a doc is deprecated.
- **Comment-inbox ritual** — start a session by reading `INBOX.md`; act, then resolve.
  Model documented just below; regenerated by `bin/sync-inbox.py`.
- **Living docs** — generated docs (stats/status) come from `bin/refresh-docs.py`;
  edit inputs, regenerate before push. **Step 6.**

## Adaptations from the original single-product template

1. Lives under `admin/docs/` (module-isolated with the admin feature that renders it).
2. Per-**project** structure (portfolio-wide), not single-product; `project` is a
   frontmatter field so the board can filter by project.
3. Scripts are **Python** (`bin/*.py`) — this repo has no Node/npm build step.
4. Coexists with the existing `docs/` KB; never clobbers it.

## Comments & the inbox loop (Step 5)

Comments on a doc don't vanish into a database you never open — they surface in
`INBOX.md`, the one file you read at the start of each session.

Each comment carries:

| field | values / meaning |
|---|---|
| `id` | stable comment id (`C-###`) |
| `doc` | the doc it's on (a doc `id` like `PRD-001`) |
| `kind` | `question` · `change_request` · `note` |
| `author` | who left it |
| `status` | `open` → `in_review` → `resolved` |
| `created` | ISO timestamp |
| `resolution` | when resolved: `{ note, link }` — a one-line why + a commit SHA or doc id |

**The loop:** comments live in `admin/docs/_comments.json` (a local stub today —
`bin/sync-inbox.py` has a TODO to point it at a real store later). Run:

```
python3 bin/sync-inbox.py
```

It regenerates `INBOX.md`: **change requests first**, then questions, then notes,
newest first; resolved items drop off with a short "recently resolved" tail. The
ritual — read INBOX, act, then set the comment's status to `resolved` with a note —
is recorded in `CLAUDE.md` (Step 8) so every future session follows it.
